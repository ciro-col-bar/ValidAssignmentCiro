| Field | Detail |
|---|---|
| **ID** | BUG-003 |
| **Title** | Progress bar freezes at "100% Complete" for ~35 seconds with no user feedback when parsing a large script on the Stock Footage page |
| **Priority** | High |
| **Severity** | Major |
| **Status** | Open |
| **Module** | Stock Footage → Script Parser → Progress Indicator |
| **Reported By** | Manual exploratory session / user-reported |
| **Date** | 2026-03-18 |

---

### Environment

- **Browser:** Chromium (mcp-web-inspector / Playwright automation)
- **URL:** `https://previz-engine-m1mm9ayva-valid.vercel.app/stock-footage`
- **User Role:** Anonymous (cookie-based UUID, no login required)
- **OS:** Windows 11
- **Script Input Size:** 6,339 characters — 16-scene screenplay

---

### Preconditions

- User is on the Stock Footage page (`/stock-footage`)
- Script textarea (`#script-text`) is empty
- No prior parse session is active

---

### Steps to Reproduce

1. Navigate to `https://previz-engine-m1mm9ayva-valid.vercel.app/stock-footage`
2. Paste a large multi-scene script into the **"Script Text"** textarea (`#script-text`) — minimum ~6,000 characters with 10+ distinct scenes (e.g., a 16-scene screenplay)
3. Click the **"Parse Script"** button
4. Observe the progress indicator

---

### Expected Result

The progress indicator advances from 0% to 100% smoothly. When processing is complete (100%), the UI immediately transitions to show the parsed shots — or displays a status message such as *"Finalising shots…"* / *"Processing complete"* to indicate that work is still ongoing. The user is never left looking at a frozen "100% Complete" screen with no actionable feedback.

---

### Actual Result

The progress bar reaches **"100% Complete"** and then **freezes** for approximately **35 seconds** with no further user-facing feedback. The loading copy continues to read:

> *"Our AI is analyzing your scene and creating a professional prompt..."*

This message is stale and inaccurate at the 100% mark — the AI analysis is done, but the UI does not reflect the transition to post-processing. The user has no indication that:
- The operation is still running
- How much longer they need to wait
- Whether the application has crashed or hung

After ~35 seconds of silence, the parsed shot results (16 shots for this input) finally appear without any transition or success confirmation.

**Console evidence (Mixpanel analytics):**
```
[log] Mixpanel track: stock_footage_script_parse_start
      Properties: {script_length: 6339, script_preview: "INT. ABANDONED WAREHOUSE - NIGHT..."}
```

> **Note:** Only a `stock_footage_script_parse_start` event is fired. There is no corresponding `script_parse_complete` or `shots_generated` event observable in the console — suggesting the backend processing phase after the 100% mark is either not instrumented or takes significantly longer than the progress indicator implies.

### Impact Analysis

- **User impact:** Users who submit large scripts (6,000+ characters / 10+ scenes) are left staring at a frozen "100% Complete" progress screen for ~35 seconds with no feedback. Without any signal that the app is still working, users are likely to assume the app has crashed and attempt a page refresh or re-submit — potentially triggering duplicate API calls and additional costs.
- **Data/business risk:** Silent freezes at a completed-looking state erode user trust in the tool. For a workflow tool used by directors and producers, unpredictable hangs on core actions are a significant usability failure.
- **Violates:** US-06 Acceptance Criteria — *"Clicking 'Parse Script' triggers AI-based parsing"* (implied: with visible, accurate progress feedback throughout); TC-SF01 — *"The system processes the script; individual shots are extracted and displayed or queued for generation"* (the transition from parsing to results is broken).

---

### Console / Network Evidence

```
[log] Mixpanel track: stock_footage_script_parse_start
      Properties: {script_length: 6339}
```

- No `script_parse_complete` or equivalent event observed after the 100% freeze
- No failed network requests observed during the freeze period
- The freeze appears to be a **UI state management gap**: the progress indicator reaches 100% before the backend shot-generation phase completes, leaving the UI in a stale "done but not done" state

---

### Root Cause Hypothesis

The progress bar is driven by a synthetic front-end timer that reaches 100% before the actual API response (shot list) is received. Once the timer completes, the component enters an indeterminate state — it shows "100% Complete" but has no UI branch to handle the gap between "progress animation done" and "data received". The app stays frozen in this gap until the API response arrives, at which point it renders the shots.

**Suggested fix direction:**
- After reaching 100%, update the loading copy to *"Finalizing shots…"* or equivalent so users know processing continues
- Alternatively, cap the progress bar at 95% until the API response is received, then jump to 100% and immediately transition — keeping the displayed completion truthful
- Add a `script_parse_complete` analytics event and a UI success/transition state so the flow is fully instrumented
